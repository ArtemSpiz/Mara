## Нумерація

- **Ітерація `N`** — логічний блок робіт + своє тестування в кінці.
- Підтаски: **`N.MM`** (наприклад `3.04` — четверта таска в ітерації 3). У промпті пиши саме ці коди.

У кожному промпті до агента:

```text
Контекст: Mara / docs/TASKS.md
Виконано: 1.01, 1.02 | У роботі: 1.03
```

Після **останньої таски ітерації N** — пройти **«Тестування після ітерації N»**, лише потім переходити до N+1.

**Lighthouse / PageSpeed** свідомо перенесені в **ітерацію 7** — після повної реалізації фіч; ранні заміри без фінального коду малоінформативні.

### Про «синхронізацію з .cursor/plans»

У проєкті може лежати **окремий** автогенерований план Cursor (`*.plan.md`). Він **не оновлюється** сам, коли ти правиш цей файл. **Синхронізація** означала б: вручну підкрутити той план, щоб номери збігалися з `TASKS.md`. **Це не потрібно**, якщо ти ведеш роботу з **одного** джерела — цього файлу.

### Акцент: гнучкі блоки для нових сторінок

Блоки в Payload + Next мають бути **максимально гнучкими**, щоб **нова сторінка = набір блоків у CMS**, а не новий унікальний шаблон у коді.

| Принцип | Що робити |
|--------|-----------|
| **Один layout builder** | Одне й те саме поле `blocks` (або спільний масив типів) для Pages, лендінгів, за потреби Posts — одна схема в адмінці. |
| **Реєстр на фронті** | `BlockRenderer` + мапа `blockSlug → Component`; новий блок = новий запис у реєстрі + компонент + визначення в Payload (без розгалуження всіх сторінок). |
| **Опційність** | У блоків — опційні поля (варіант відображення, відступи, `anchor`, фон), щоб комбінувати без нових типів. |
| **Чисті props** | Компонент блоку приймає серіалізовані дані (без документів Payload у JSX); мапінг URL медіа в одному шарі. |
| **Документація** | Короткий `docs/cms-blocks.md`: кроки «як додати блок» для розробника. |

Ці принципи закріплені тасками **5.11–5.14** нижче.

---

### Акцент: жодних захардкожених текстів і «налаштувань» у коді

Увесь **видимий користувачу текст** і **структуровані дані, що впливають на сайт** (підписи меню, форми, кнопки, hero, футер, legal, порожні стани) мають **жити в Payload** (globals + блоки + колекції), а не в JSX/TS як літерали. У коді допустимі лише: ключі, типи, мапінг полів CMS → компонентів, технічні дефолти для preview.

Технічні винятки (узгоджувати явно): `aria-*` / порожній `alt` для декоративних елементів — або з CMS, або з **одного** модуля констант, який за потреби теж заповнюється з API.

Таски **5.15–5.20** нижче.

### Дизайн-токени: Tailwind (v4)

У проєкті **Tailwind v4**: глобальна тема задається в **`app/globals.css`** у блоці **`@theme`**, файл **[tailwind.config.mjs](../tailwind.config.mjs)** підключений через **`@config`** (плагіни, розширення). Опис для команди — [docs/tailwind.md](tailwind.md).

| Правило |
|--------|
| Нові **кольори** — тільки як `--color-mara-*` у `@theme`, у розмітці `bg-mara-*` / `text-mara-*` / `border-mara-*`. |
| Нові **розміри шрифтів** (якщо не вистачає стандартного scale) — `--text-mara-*` у `@theme`. |
| **Шрифти** (сімейства) — `next/font` + прив’язка в `@theme`; не дублювати ім’я шрифту в класах крім утиліт. |
| Уникати нових arbitrary `text-[#...]`, `bg-[#...]` після міграції на токени (ітерація **3B**). |
